# spec-distill

> **配置说明**：vault 的结构配置存在**家目录注册表** `~/.config/obsidian-wiki/configs/<库名>.json`（按 active 库解析；未注册则回退库内 `<vault>/.wiki/config.json`）。schema 见本插件根 `config.example.json`。脚本仍通过 `--vault "<vault 根路径>"` 指定 vault，结构由注册表/回退提供。
>
> **脚本定位（插件）**：脚本随插件安装，运行前先解析插件根 `$WIKI`：
> ```bash
> WIKI="${CLAUDE_PLUGIN_ROOT:-${CODEBUDDY_PLUGIN_ROOT:-}}"; [ -d "$WIKI/skills/spec-distill" ] || WIKI="$(find "$HOME/.claude/plugins/cache" "$HOME/.codebuddy/plugins/cache" "$HOME/.copilot/installed-plugins" -type d -path '*/obsidian-wiki/skills/spec-distill' 2>/dev/null | sort -V | tail -1 | sed 's:/skills/spec-distill$::')"
> ```
> 下文命令里写的 `scripts/kn_scan.py` 一律指 `"$WIKI/skills/spec-distill/scripts/kn_scan.py"`。

从 SpecIn 里 specode 生成的需求文档，**逐项目**提炼出**细粒度知识文档**，落到
`10-Work/知识库/<系统>/`，并维护该系统 `MEMORY.md`。
本 skill **替换原 `kn-indexer`（含 Windows 端），跨平台**。

---

## 命令表

| 命令 | 行为 |
|---|---|
| `/spec-distill`（= `scan`） | 跑 `python3 scripts/kn_scan.py scan`，读 `00-Index/_system/spec-distill-report.md`，汇报待沉淀清单并提议先做哪个；**不写知识文档** |
| `/spec-distill scan [--source <路径>] [--vault <路径>]` | 同上，可手动指定源目录或 vault 根 |
| `/spec-distill sync [--source <路径>] [--project <名>] [--rescan] [--dry-run]` | 逐项目沉淀（见"sync 逐项目流程"），LLM 驱动，用户控节奏 |
| `/spec-distill help` | 显示本命令表与流程说明 |

> **重要区分**：`scan` 是脚本子命令（`kn_scan.py scan`），可直接运行；
> `sync` **不是**脚本子命令——它是本 SKILL.md 编排的 LLM 交互工作流，
> 不得向脚本传入 `sync` 参数。

未给子命令时按 `scan` 处理。

## scan 流程

1. 运行脚本：`python3 "$WIKI/skills/spec-distill/scripts/kn_scan.py" scan --vault "<vault>" [--source <路径>]`
   - `--vault` 必填（脚本经 `load_config` 按该路径在家目录注册表取配置，未注册则回退 `<vault>/.wiki/config.json`）。
   - 源目录默认 `<SpecInRoot>/windows-Public/specs/`；`SpecInRoot` 自动探测 `SpecIn/`（无则 `spec-in/`），可由配置 `knowledge.spec_in_candidates` 改。
2. 读报告 `00-Index/_system/spec-distill-report.md`（脚本写出）。
3. 向用户汇报：
   - 待沉淀项目列表（项目名 + key）。
   - 已有系统与已覆盖需求号数量。
   - 提议优先处理哪个项目（按工作项目特征判断，跳过个人/测试项目）。

## sync 逐项目流程

对每个待沉淀项目，按以下六步顺序执行。**不可省略任何一步。**

### 第 1 步：AskUserQuestion — 选系统归属

- 动态读 `10-Work/知识库/` 现有子目录列表，逐一列出。
- 追加选项：`新建系统`、`跳过本项目`。
- 若用户选"新建系统"：再问目录名（如 `财务中台（fap）`）→ `mkdir 10-Work/知识库/<系统名>/`
  → 用 `references/memory-rules.md` 中"新系统骨架"建空 `MEMORY.md`。
- 若用户选"跳过"：在 `00-Index/_system/wiki-log.md` 追加一行
  `- YYYY-MM-DD HH:MM spec-distill SKIP <项目名>`，然后进入下一项目。

### 第 2 步：读全项目

- Read 该项目目录下所有 `.md` 文件：
  `requirements`、`design`、`tasks`、`implementation-log`、`bugfix`、
  `acceptance-checklist`、测试报告等。
- 子目录限 3 层深。

### 第 3 步：AskUserQuestion — 提议知识点拆分（不可跳过）

- 按 `references/breakdown-heuristics.md` 中的启发式，输出 N 个候选知识点，
  每个包含：拟标题、一句摘要、拟 tags。
- **AskUserQuestion 让用户确认/增删/合并/改名**，拿到最终列表后再进行下一步。
- 不得自动跳过此确认步骤。

### 第 4 步：逐篇写深度知识文档

- 每个用户确认的知识点，按 `references/doc-template.md` 模板写一篇 `.md`，
  落到 `10-Work/知识库/<系统>/`。
- 深度对齐现有语料：含表名、字段、调用链、分库分表、文件路径、枚举值、坑点。
- 文件名冲突（同系统已有同名）：加 `-v2`/`-v3` 后缀，并向用户报告。

### 第 5 步：更新 MEMORY（只追加，不重排）

按 `references/memory-rules.md` 中的更新规则：

- "知识文档索引"：每篇新文档追加一行 `| [[<文档名>]] | <一句摘要> | <需求号> |`。
- "需求反向索引"：该需求号追加/更新一行（关联知识列列出本次新建文档）。
- "知识关联图"：按用户确认追加节点，其余不动。
- 不重排旧行，不改 H1，不改已有图节点。

**MEMORY 头部（首次 sync 到某系统时）**：

将头部重写为：

```text
> 区域索引：[[10-Work/README|10-Work]]

# Knowledge Base Index — <系统名>

> Auto-generated by spec-distill. 手动编辑请保留表格结构。
> 知识库路径: 10-Work/知识库/<系统名>
```

同时删除原有 `> Vault: C:/Users/…` 这类写死某台机器路径的行（跨平台兼容）。
详见 `references/memory-rules.md`。

### 第 6 步：写 wiki-log

向 `00-Index/_system/wiki-log.md` 追加一行：

```text
- YYYY-MM-DD HH:MM spec-distill <项目名> -> 10-Work/知识库/<系统>/ (需求号: X, 新建: N 篇)
```

**不动 SpecIn 原文。**

## 知识点拆分启发式

详见 `references/breakdown-heuristics.md`。

核心原则：一个知识点 = **一个可独立复用、可被 `[[..]]` 引用的结论单元**。
避免太碎（一个字段单独成篇）也避免太粗（整项目一篇）；优先抽"会被多个需求复用"的稳定结论。
拆分由 LLM 提议、用户拍板（第 3 步）。

## 红线

以下规则**不可绕过**：

| 红线 | 说明 |
|---|---|
| SpecIn 只读 | 永不修改/移动/重命名 SpecIn 下任何文件 |
| 唯一可写口袋 | 只写 `10-Work/知识库/<系统>/`；其余 `10-Work/` 子目录（权限申请/系统/运维助手/记录）只读 |
| 敏感信息拦截 | 遇账号/明文 token/密钥/人员姓名+联系方式 → AskUserQuestion 列出原文位置，让用户决定纳入/脱敏/跳过 |
| 改前必读 | 写任何现有文件前先 Read 其全文 |
| 批量写前备份 | 同系统批量写文档前，tar 备份该系统目录到 `~/Library/Caches/spec-distill-backup-<ts>/`（Windows 退化为同盘临时目录） |
| 写后追加日志 | 每次写操作后 append `00-Index/_system/wiki-log.md` |
| 外置盘检查 | vault 路径含 `/Volumes/` 时，先确认挂载（`ls "/Volumes/External HD"` 能访问）；失败则报错停止，不静默写到其他位置 |
| 不联网 | 本 skill 全程不访问网络 |

## 跨平台

- 所有路径相对 vault 根操作，不硬编码绝对路径。
- 源目录自动探测 `SpecIn/`（无则 `spec-in/`），兼容 Windows 上仍叫 `spec-in/` 的库。
- 脚本 `kn_scan.py` 使用 Python 标准库、UTF-8 读写；不依赖外部包。
