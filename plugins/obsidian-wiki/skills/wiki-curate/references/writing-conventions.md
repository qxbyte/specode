# 写作规范

`wiki-curate` skill 在创建或更新笔记时遵循的完整写作规范——目录模型、笔记类型、命名、标签、frontmatter、双链。

## 职责边界

本 skill（wiki-curate）只改**内容笔记正文与归属目录**：

- `ingest`/`curate` 修改内容笔记的用途段、标签、正文双链，以及把文件放入对应 Wiki 目录（用户确认后）。
- `lint` 报告内容健康问题（缺用途、重复 basename、孤儿、frontmatter 缺字段、过时声明、矛盾对）。

以下内容**不在本 skill 职责范围内**：

- **index/分区页/Home 由 wiki-struct 受管块维护**：`00-Index/Home.md`、各一级目录 `README.md`、`00-Index/<目录>.md` 分区页的 `<!-- wiki-struct:tree ... -->` 块归 wiki-struct，wiki-curate 不手改。
- **`10-Work/知识库/` + MEMORY 是遗留产物（只读）**：`10-Work/知识库/<系统>/` 下的知识文档与 `MEMORY.md` 是已剥离的 spec-distill（v2.0.0 迁往 specode 的 `/specode:distill`）的历史产物，wiki-curate 不写该子树。
- **坏链与结构漂移由 wiki-struct 负责**：broken link、分区页完整性检查见 `/wiki-struct check`。

---

## 总原则

- **中文优先**：目录说明、笔记标题、标签、摘要、用途段一律中文。
- **保留业界通用英文词**：LLM、RAG、API、MCP、SDK、CLI、agent、workflow、embedding、wiki、Claude Code、Obsidian、OpenAI、Anthropic、Karpathy 等。
- **每篇笔记必含"用途"段**：`01-Concepts/` ~ `08-Sources/` 下的笔记必须有 `## 用途`，说明它解决什么问题、应该从哪些索引进入。
- **双链优于硬路径**：所有内部链接用 Obsidian wikilink `[[笔记]]` 或 `[[路径/笔记|显示名]]`，**绝不**用 markdown link `[文本](路径.md)`——后者会绕过反链系统。
- **来源外部链接**用普通 markdown link，并尽量注明访问日期。
- **保护已有内容**：修改前先 Read 相关文件；避免无关重排和大面积改写。

## 目录模型

```text
00-Index/      全局入口、分区页、_system 机器可读文件（结构由 wiki-struct 管）
01-Concepts/   稳定概念和机制：RAG、工具调用、上下文窗口、MCP、事件溯源、Saga
02-Models/     模型家族、具体模型、上下文窗口、模态、限制（如 Claude Opus 4.7、GPT-5）
03-Memo/       私人运维备忘（服务器、激活码、账号、密钥）——敏感，仅路径级引用
04-Tools/      CLI、SDK、MCP server、向量数据库、评估框架、agent 框架（Claude Code、Obsidian）
05-Workflows/  可复用流程、操作步骤、决策树、检查清单、常见故障
06-Prompts/    可复用 prompt、提示词模式、评估提示词、agent 协作模板
07-Ideas/      项目想法、产品需求、技术探索——硬只读 Raw source
08-Sources/    论文、官方文档、changelog、发布说明、文章的资料卡片
09-Journal/    按日期记录的观察、实验过程、临时想法和每日整理
10-Work/       工作资料、需求项目、知识库——硬只读 Raw source（知识库为遗留 spec-distill 产物，只读）
99-Inbox/      待归档材料、临时摘录、尚未判断归属的内容（可流转）
Clippings/     网页剪藏暂存（可 ingest 后归档）
Database/      DB Folders 插件高级元数据视图
SpecIn/        Specode 工作流规格文档——硬只读 Raw source
_scaffold/     笔记模板骨架（不计入孤儿检测）
```

共 16 个顶级目录。每个 Wiki 目录都有 `说明.md` 解释作用；本 skill 在新建分类时同步创建 `说明.md`。

## 笔记类型

| 类型 | 典型位置 | 特征 |
|---|---|---|
| **索引笔记** | `00-Index/*.md` | 导航、主题地图、内容入口、相关笔记列表 |
| **概念笔记** | `01-Concepts/` | 一个稳定想法或机制，一篇只讲一个核心概念 |
| **实体笔记** | `02-Models/`、`04-Tools/` | 一个模型、库、CLI、API、产品 |
| **工作流笔记** | `05-Workflows/` | 可重复执行的步骤、命令、检查清单、失败模式 |
| **提示词笔记** | `06-Prompts/` | 可复用 prompt、prompt 模式 |
| **来源笔记** | `08-Sources/` | 对论文、文档、版本发布、文章的摘要和引用 |
| **日记笔记** | `09-Journal/` | 按时间沉淀的学习、实验、观察 |
| **衍生页** | `00-Index/`、`01-Concepts/` 等 | 基于 07-Ideas/10-Work/SpecIn 中只读原文提炼出的 Wiki 区页面 |

## 命名

- 中文加必要英文名：`工具调用 Tool Calling.md`、`Claude Code.md`、`RAG 评估.md`
- 项目级索引用 `MEMORY.md` 或 `README.md`（保持与 vault 现有风格）
- 衍生页文件名沿用原文的核心词：`07-Ideas/CodeMap/README.md` → `01-Concepts/代码图谱.md`
- 不要带特殊字符 `[ ] | # ?`；空格、`-`、中文括号 `（）` OK

## 标签

- **说明性标签优先中文**：`概念`、`模型`、`工具`、`架构`、`工作流`、`评估`、`来源`、`日记`、`提示词`
- **命名空间标签保留**：`LLM/wiki`、`llm/wiki` 适合检索分组
- **不要无理由新增大量标签**：优先少而稳定
- **整理时**：发现旧英文标签（`architecture`、`daily`）可改为或补充中文（`架构`、`日记`），不要破坏现有检索

## frontmatter

每篇 Wiki 笔记的 frontmatter 必含：

```yaml
---
类型: 概念        # 概念/模型/提供方/工具/工作流/提示词/评估/来源/日记/索引/衍生
状态: 草稿        # 草稿/稳定/待复核/已废弃
标签:
  - LLM/wiki
  - 概念
别名: []          # 可选，写作"中文别名"或"English Alias"
更新: 2026-06-05  # ISO 日期
---
```

衍生页额外字段：

```yaml
来源: "[[07-Ideas/CodeMap/README]]"   # 原文位置
```

## 笔记小节顺序

```markdown
# 笔记标题

## 用途

（必填）这篇笔记解决什么问题、服务什么场景、应该从哪些索引进入。

## 摘要

（必填）两到四句话说明核心观点。要写成以后搜索时能直接读懂的答案。

## 说明

- 关键事实或定义
- 实践意义
- 限制、风险或容易误解的地方

## 关联

- 相关：[[另一篇笔记]]
- 来源：[官方文档](https://example.com)（访问日期：2026-06-05）
```

工作流笔记额外加 `## 步骤` 和 `## 失败模式`；来源笔记额外加 `## 引用` 和 `## 我的批注`；评估笔记额外加 `## 维度` 和 `## 结果`。完整模板见 `note-templates.md`。

## 双链规则

- **内部链接**：`[[笔记标题]]`、`[[路径/笔记标题|显示名]]`。Obsidian 会用 fuzzy 匹配，但**有歧义时优先写完整路径**。
- **绝不用 markdown link 做内部链接**：`[文本](笔记.md)` 会绕过反链。
- **在表格内的 wiki link 用 `\|` 转义 alias 分隔符**：`| [[路径/笔记\|显示名]] | 摘要 |`，否则 `|` 会被当成表格分隔符。
- **每篇新笔记至少被一个索引或相邻主题链接**：避免孤儿页。
- **追加段落里加双链**：例如在概念笔记里追加"实践案例"小节，引用 `[[07-Ideas/CodeMap/README]]`。

## 索引页要求

索引页**不应只是文件列表**。每个条目要有：

- 链接 + 一句话摘要
- 可选元数据：关键词、可以回答的问题、状态
- 必要的敏感内容处理说明（针对 10-Work、99-Inbox 等）

示例（`00-Index/04-Tools.md` 一行）：

```markdown
- [[04-Tools/Obsidian]] — 本地优先的双链笔记软件，本 Wiki 的 IDE；可回答 "Obsidian CLI 怎么用"、"DB Folders 插件配置"
```

## 策展流程

### 新主题

1. 用 grep 和 Read 收集本地上下文。
2. 先更新相关索引，让主题有入口。
3. 按需创建概念、实体、工作流、来源、日记等原子笔记。
4. 每篇新笔记写"用途"段。
5. 添加必要的双向链接、来源链接、中文摘要。
6. 最后检查坏链、重复标题、frontmatter 不一致、缺少索引入口。

### 已有主题

1. 修改前阅读该笔记和相邻笔记。
2. 只有确实同义才合并重复页。
3. 移动或重命名时保留别名和重要反链。
4. 时间线重要时增加日期段落；否则自然合并进正文。
5. 英文说明性标签、英文目录说明、缺用途的笔记，优先中文化和补齐。

### 只读三目录的特别流程

参见 `readonly-dirs-policy.md`。要点：

- 衍生页放 Wiki 区，原文不动
- 反向链接放 `00-Index/_system/backlinks-readonly.md`，不写入原文
- 敏感子目录只到路径级，不进入正文摘要

## 验证

完成前尽量检查：

- 用 `rg -n "\[\[[^\]]*$" <vault>` 找没闭合的 wikilink
- 新建多篇笔记时检查重复标题（同 basename 在多目录）
- 每篇新笔记至少被一个索引或相邻主题链接
- 新建目录应有中文说明页或索引页
- 新建笔记应包含"用途"段
- 如果命令无法运行，说明跳过了什么以及原因

完整 lint 清单见 `lint-checklist.md`。
